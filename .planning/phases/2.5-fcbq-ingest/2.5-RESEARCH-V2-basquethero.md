# Phase 2.5 — basquethero.cat Source Discovery Spike (REVISED)

> **Read this AFTER `2.5-RESEARCH.md` (the v1 FCBQ STOP report).** Roger's decision 2026-05-22 (AFK overnight) was to switch the source from `basquetcatala.cat` (blocked by reCAPTCHA v3) to **`basquethero.cat`**, a public Catalan basketball stats aggregator built on the same FCBQ data.

---

## 🟡 Executive Summary

`basquethero.cat` is a **public Next.js + RSC site on Vercel** that aggregates FCBQ data for free. Key differences from FCBQ:

| Dimension | FCBQ (`basquetcatala.cat`) | basquethero.cat |
|---|---|---|
| Accessibility | 🛑 reCAPTCHA v3 wall on every URL | ✅ Open. Permissive `robots.txt`. |
| Roger's league slug | `cc-2a-m` mapped to internal int `19773` (must search behind wall) | ✅ Direct slug: `/liga/cc-2a-m-grup-01` (01..06 groups) |
| Calendar accessible | Behind wall | ✅ `/liga/<slug>/calendario` returns 200 |
| Data rendering | Server-rendered HTML (calendar + standings) + JPG actas + React SPA stats | Next.js App Router + React Server Components — data streamed via `self.__next_f.push([N, "<payload>"])` chunks |
| Player stats | Behind RS256 JWT to third-party `msstats.optimalwayconsulting.com` | ✅ Likely embedded in RSC payloads (`/liga/<slug>/jugadores`) |
| Stdlib-only parser feasibility | ❌ Impossible (captcha solver required) | 🟡 Possible but **medium complexity** — RSC payloads are React JSX trees, not clean data dicts |

**Verdict:** the LOCKED architectural constraints (httpx-only, zero new deps, stdlib-feasible parser) **HOLD** with basquethero.cat. However, the parser is NOT `bs4.select("table tr")`. It's a **RSC-payload walker** (~200-400 lines of stdlib Python: `urllib` + `re` + `json` + recursive tree walk). Manageable for a portfolio project; needs explicit allowance in PLAN.md for "non-trivial parser inside ingest/".

### 6. `parser_lib_recommendation`

**Decision: stdlib-only — feasible, medium complexity.**

| Approach | Status | Notes |
|---|---|---|
| `httpx` + `bs4` (D2.5-02 / D2.5-09 default) | ⚠️ Insufficient | `bs4` can extract dates from HTML text, but team names + game IDs are encoded inside RSC chunks (`self.__next_f.push([N, "..."])`) — they look like React JSX tree props, not clean DOM tables. |
| **stdlib `urllib` + `re` + `json` + recursive walker** | ✅ Recommended | Spike confirmed: 165 RSC chunks extracted, 134 decoded as JSON. Parser walks tree looking for `<a href="/equipo/<id>">` link nodes + adjacent text nodes. ~200-400 lines. |
| `httpx` + `bs4` + `selectolax` for fast HTML | ⚠️ Optional but doesn't change picture | If `bs4` is allowed for the HTML parts (dates, basic shell), selectolax adds 1 dep. NOT recommended — sticks with stdlib. |
| Playwright (headed Chrome) | ❌ Not needed | Was the STOP trigger for FCBQ. NOT required here — content is in the initial HTML response stream. |
| Open Data FCBQ paid feed | ❌ Not relevant | Would still work but is no longer needed. |

**Implication for D2.5-09 (zero new top-level deps):** holds. Parser is stdlib-only (urllib, re, json — all built-in to Python 3.12+).

**Implication for D2.5-02 (httpx-only):** can be downgraded to "urllib" for ingest path — fewer deps. OR keep httpx for consistency with API runtime path, and use it in ingest too (httpx is already a dep elsewhere). Either is acceptable. Recommend: **`urllib` in ingest (zero deps), `httpx` retained for API runtime**.

### 9. `gotchas` (priority order)

1. **🟡 RSC tree is JSX, not data.** The parser walks React component trees, not entity dicts. Each "game" appears as a `<li>` or `<div>` with `children: ["27 sep", {component: a, props: {href: "/equipo/123", children: "Team A"}}, ...]`. Pattern-matching, not key-lookup.
2. **🟡 Page weight: 1.3 MB per calendar page.** Single league = single fetch, but if ingesting all 6 cc-2a-m groups = ~8 MB. Acceptable for offline batch (D2.5-04 offline batch policy).
3. **🟢 No anti-bot signals beyond standard UA convention.** Sustained scraping untested but no evidence of rate limiting in initial probes.
4. **🟢 Data freshness:** unknown. basquethero aggregates from FCBQ on a schedule (likely daily). Won't have live-game updates but Phase 2.5 ingest is for historical / completed games anyway.
5. **🟡 Slug discovery still needed for some leagues.** Roger's cc-2a-m is direct, but if Roger plays in a sub-group not visible in sitemap, manual map needed.
6. **🟢 Sitemap is monolithic 3.7 MB.** Fetch once, cache, parse on-demand.
7. **🟢 robots.txt explicitly Allow /.** Legal/ethical: clear permission.
8. **🟡 No `/_next/data` page-data endpoint exposed.** Would have been the cleanest path but it's disabled. RSC walking is the only option.
9. **🟡 Player-level stats not yet probed.** Likely at `/liga/<slug>/jugadores` — same RSC pattern. Wave 1 task to verify.

## Spike Artifacts

- **`spike-basquethero/spike.py`** — 130-line stdlib-only Python 3.12+ script.
  - Fetches `/liga/<slug>/calendario` via `urllib.request`.
  - Extracts RSC chunks via regex.
  - Decodes JSON payloads.
  - Walks tree for game-like nodes (heuristic markers).
  - Prints top dict-key frequency.
  - Exit codes: 0 = game node found, 1 = no game node but RSC OK, 2 = network fail, 3 = no RSC chunks (page format changed).
