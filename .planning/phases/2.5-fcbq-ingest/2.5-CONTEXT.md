### D2.5-02 — HTTP client: `httpx` async
**Decision:** Use `httpx.AsyncClient` for all FCBQ fetches. No new top-level dep (`httpx` is already a transitive dep via FastAPI/Starlette tests).

**Why:**
- Codebase is async end-to-end (SQLAlchemy async, asyncpg, FastAPI). Using sync `requests` would force `asyncio.to_thread` wrappers — friction without benefit.
- Streaming + native timeouts + connection pooling are first-class.
- Same client lib for both runtime tests and the CLI keeps mental overhead low for reviewers.

**Rejected:** `requests + tenacity` (sync, breaks codebase consistency), `playwright` (only revisit if D2.5-01 spike reveals JS-rendered pages with no JSON XHR fallback).

---

### D2.5-03 — Two-phase CLI: `fetch` then `parse` with raw cache (gitignored)
**Decision:** Split the CLI into two sub-commands that can run independently:
- `fcbq fetch <comp> <season>` → writes raw bytes to `data/raw/fcbq/<comp>-<season>/<resource>.{html,json}`. Idempotent: skips files already present unless `--force`.
- `fcbq parse <comp> <season>` → reads raw cache, validates with Pydantic, writes normalised `data/seed/fcbq/<comp>-<season>.json`.
- `fcbq scrape <comp> <season>` → convenience wrapper for fetch + parse end-to-end.

`data/raw/` added to `.gitignore`. Only the normalised `data/seed/fcbq/*.json` is committed.

**Why this is the production-grade choice:**
- **Courtesy to FCBQ:** while iterating on the parser, zero re-HTTP. Public site, non-commercial — but still a real org with real bandwidth.
- **Resumability:** if the scrape interrupts at game 47/120, the next `fetch` run resumes from where it stopped.
- **Offline debuggability:** when a parser breaks because FCBQ changed a CSS class, you can rerun `parse` against the cached raw without re-fetching.
- **Recruiter signal:** ETL separation-of-concerns (Extract → Transform → Load) is what production data pipelines look like. A monolithic `scrape → write` script reads as junior; this reads as senior.
- **Diff hygiene:** raw HTML is noisy. Keeping it out of git makes the PR diff small and reviewable.

**Rejected:** Single-pass in-memory (every re-run hits FCBQ), `--cache-raw` opt-in flag (more code paths, less consistent).

### D2.5-04 — Fixture shape: 1 file per (comp, season), top-level + nested children
**Decision:** Each committed fixture is one JSON file with this shape:
```json
{
  "fixture_version": "1",
  "scraped_at": "2026-05-21T12:00:00Z",
  "source_url": "https://www.basquetcatala.cat/...",
  "competition": { "slug": "cc-2a-m", "season": "2025-26", "name": "...", ... },
  "teams": [ { "fcbq_team_id": ..., "club_slug": ..., "display_name": "CB Granollers", ... }, ... ],
  "players": [ { "license_id": 12345, "first_name": "...", "last_name": "...", "position": "PG", ... }, ... ],
  "games": [
    {
      "fcbq_game_id": ..., "date": "2025-10-04", "home_team_id": ..., "away_team_id": ...,
      "q1_home": 18, "q1_away": 22, ..., "q4_home": ..., "q4_away": ...,
      "box_scores": [ { "player_license_id": 12345, "minutes": ..., "fg2m": ..., ... }, ... ]
    }, ...
  ]
}
```

**Why:**
- One file ↔ one transaction at load time. Simple loader, atomic.
- Self-contained fixture: anyone reading the file sees the full competition without joining 5 files.
- Diff-friendly when a single value changes (e.g., FCBQ corrects a stat) — the rest of the file is unchanged.
- `fixture_version` + `scraped_at` + `source_url` are audit metadata — recruiter sees provenance discipline.

**Rejected:** Per-entity split (`competitions.json`, `teams.json`, etc. — relational but loaders coupled by FK ordering); per-game files (filesystem explosion, no way to make load atomic across the comp).

**File size budget:** Catalan amateur leagues run ~12-16 teams × 22-30 games/season × ~12 box-scores/game ≈ ~5000 box-score rows per fixture. JSON ~2-5 MB uncompressed. Acceptable for git.

### D2.5-05 — Player natural key: `license_id` (numeric, FCBQ federation-issued)
**Decision:** Player.`license_id` (BIGINT UNIQUE NOT NULL) is the natural key for UPSERT and for cross-fixture deduplication. UUID `id` remains the surrogate PK (already in P2 migration).

**Why:**
- Matches the P2 D2-decision ("Player PK híbrid"). Zero conceptual drift.
- FCBQ assigns one license per (player, season) — globally unique across all Catalan competitions. Lookup by `license_id` is O(1) and never ambiguous (unlike name+birthdate which has homonym risk).
- Players who change clubs keep the same `license_id` → cross-comp identity preserved → leaderboards aggregating across competitions work correctly.

**Edge case to handle in parser:** Some young / amateur leagues may show players without published license IDs (privacy or scraping limitation). Decision: parser SKIPS that row + emits a structured warning to stderr. The fixture excludes them. Better to under-cover than to invent IDs that collide later.

**Rejected:** Hash of (name + birthdate) — name collisions in Catalan basketball (multiple "Marc Soler"s) are real; birthdate often absent in public listings.

### D2.5-06 — Loader idempotency: `INSERT ... ON CONFLICT DO UPDATE` per natural key
**Decision:** Loader uses Postgres UPSERT keyed on natural keys:
- `competition`: `ON CONFLICT (slug, season)` — competitions are append/update, never deleted.
- `team`: `ON CONFLICT (fcbq_team_id)`.
- `player`: `ON CONFLICT (license_id)`.
- `game`: `ON CONFLICT (fcbq_game_id)`.
- `box_score`: `ON CONFLICT (game_id, player_id)`.

Re-running the loader against an unchanged fixture is a no-op (zero rows changed). Against an updated fixture, only changed rows are touched.

**Why:**
- Idempotent. Run twice safely. Easy to retry on transient network errors during the loader run.
- Preserves any FK relationships to rows not in this fixture (e.g., manually-uploaded coaches from P3 won't be deleted).
- Postgres-native; SQLAlchemy 2 supports it via `insert(Model).on_conflict_do_update(...)`.

**Rejected:** DELETE + INSERT (cascades to dependent rows and loses any P3 augmentation); diff-based row-by-row (overkill for portfolio; UPSERT is the industry default for this exact use case).

### D2.5-07 — Initial coverage: `cc-2a-m 2025-26` ONLY for the first ship
**Decision:** First PR ships exactly one fixture: `data/seed/fcbq/cc-2a-m-2025-26.json` (2a Catalana Masculina, current season). Other competitions added in follow-up PRs (or P2.5.1 if scoped separately).

**Why:**
- The Phase 2.5 goal is **closing SC6** and proving the pipeline end-to-end, not full FCBQ coverage. 1 competition × 22-30 games is enough to make `/docs` look populated, make `GET /competitions` return non-empty, and make `/leaderboards` produce meaningful results.
- Adding all 12 (comp × gender) combinations multiplies scrape time, fixture review burden, and the risk of FCBQ rate-limiting before we've validated the pipeline once.
- Why `cc-2a-m` specifically: mid-tier competition (not Supercopa with 8 elite teams, not 2a Territorial with sparse data). Realistic data density, manageable team count.
- Why `2025-26`: current season — FCBQ tends to keep the most reliable + complete data for the ongoing season; historical seasons sometimes have gaps.

**Future iteration plan:**
- Iteration 2: add `cc-2a-f-2025-26` (women's mirror — recruiter signal: gender-balanced coverage).
- Iteration 3: add Supercopa M+F.
- Iteration 4+: territorial leagues by group (bcn / gir / tar / lle).

Each iteration = its own PR with its own fixture. Loader doesn't change; only data grows.

**Rejected:** Ship all 12 combinations at once (multiplies risk, slows first ship, hides parser bugs behind volume).

### D2.5-08 — Anti-bot / reliability: rate limit 1 req/s + tenacity backoff + UA rotation + structured failures
**Decision:**
- **Rate limit:** `asyncio.sleep(1.0 + random.uniform(-0.2, 0.5))` between requests (jittered 0.8-1.5s window). One concurrent request at a time (no `asyncio.gather`).
- **Retry:** `tenacity.retry(stop=stop_after_attempt(5), wait=wait_exponential(multiplier=2, min=2, max=30))` on `httpx.HTTPError` and 5xx responses. 4xx (other than 429) fail-fast without retry — likely a bad URL, retrying won't help.
- **UA rotation:** Pick at random from a list of 4-5 current Chrome/Firefox UA strings on macOS+Windows. Do NOT spoof googlebot (could be misread as bot abuse).
- **HTML drift handling:** Parser raises a specific `FCBQParseError(raw_path, missing_selector, expected_format)` when the expected structure isn't found. CLI prints the offending raw file path + exits non-zero. Operator inspects the cached raw to understand the change, updates the parser. Sample fixture in CI then detects schema regressions automatically.
- **Resumability state:** `data/raw/fcbq/<comp>-<season>/.scrape-state.json` tracks `{completed_resources: [...], failed_resources: {path: error_msg}}`. `fetch` skips completed; `--retry-failed` re-attempts the failed list.

**Why:**
- 1 req/s is well under what would trigger any sane rate-limiter on a public sports federation site, but slow enough that we look like a polite browser user.
- Tenacity exponential backoff with max 5 attempts is the industry default for "transient network" handling.
- Specific exception with the raw file path means debugging is 1 step (open the file, look for what changed), not 5 steps.
- Resumability is the difference between "the scrape failed at 80% so we restart from 0%" (junior) and "the scrape resumes from 80%" (senior).

**Rejected:** Aggressive 0.1 req/s parallel (rude + ToS risk), no retry (cheap network blips fail the whole run), captcha solver (out of scope — if FCBQ adds CAPTCHA we revisit with Roger).

### D2.5-09 — Module layout + CLI library: stdlib `argparse`, modular sub-package
**Decision:**
```
src/basketball_stats/ingest/
  __init__.py
  fcbq/
    __init__.py
    __main__.py          # argparse CLI: fetch | parse | scrape
    client.py            # httpx wrapper + rate limit + retry + UA rotation
    parser.py            # raw HTML/JSON → dataclasses
    models.py            # Pydantic models for the fixture JSON schema (re-used by loader)
    state.py             # .scrape-state.json read/write
    exceptions.py        # FCBQParseError + friends
data/
  seed/
    fcbq/                # committed fixtures
    load_fcbq.py         # standalone loader (sibling to minimal.py)
  raw/                   # gitignored, raw HTML/JSON cache
    fcbq/<comp>-<season>/
```

CLI lib: stdlib `argparse` (no new deps).
Logging: stdlib `logging` configured at CLI entry (`__main__.py`). No `rich`.

**Why module split this way:**
- Single sub-package `ingest.fcbq` keeps "FCBQ-specific knowledge" colocated. If we ever add a second federation (RFEB, Euroleague), it becomes `ingest.rfeb` etc. without touching FCBQ code.
- `models.py` (Pydantic) is shared between the parser output side (validate before writing JSON) and the loader read side (validate before INSERT). One source of truth for the fixture schema.
- `state.py` isolated because it's the one part with filesystem side-effects beyond raw caching — easier to mock in tests.
- `loader` lives under `data/seed/` next to `minimal.py` (existing seed convention) — NOT inside the `ingest` package. **Loader is the only bridge between offline data and production schema** — it imports from `basketball_stats.models` (allowed) but not from `basketball_stats.ingest.*` (allowed only at runtime by the operator, not by FastAPI).

**Why argparse over typer/click:**
- Zero new top-level deps. Pyproject stays clean.
- Recruiter signal: "they reach for the stdlib when it's enough" is a strong positive. Adding typer for 3 sub-commands reads as over-engineering.
- We control the CLI surface ourselves — argparse sub-parsers handle `fetch/parse/scrape` in ~30 lines.

**Rejected:** Typer (extra dep, marginal UX gain), Click (same), separate package per command file (over-engineered).

### D2.5-10 — CI gates: 3 new checks
**Decision:** Add to existing GHA workflow:

1. **Import ban grep** (new step, fails build on match):
   ```bash
   if grep -r "from basketball_stats.ingest" src/basketball_stats/ --exclude-dir=ingest; then
     echo "❌ Production code imports from offline-only ingest package"; exit 1
   fi
   ```
   Enforces LOCKED no-live-ingest at the import-graph level. Runs before tests.

2. **Sample fixture loader integration test** (`tests/integration/test_load_fcbq_fixture.py`):
   - Uses `testcontainers.PostgresContainer` (already in P2 stack).
   - Loads `tests/fixtures/fcbq/sample-mini.json` (1 competition + 2 teams + 1 game + 12 box-scores — same shape as production fixtures, minimal size).
   - Asserts: row counts match expected, all FKs resolve, VAL GENERATED column populated for all box-scores.
   - **Idempotency assertion:** load fixture twice in the same test → second load returns same row counts (no duplicates, no errors).

3. **Schema validation** (part of `parser.py` unit tests): load `sample-mini.json` through `models.py` Pydantic models and assert no validation errors. Catches fixture-schema drift before integration tests do.

**Why:**
- Import-ban grep is the cheap, definitive enforcement of the LOCKED constraint. Doc-level promises rot; CI checks don't.
- Sample fixture (not the real `cc-2a-m-2025-26.json`) keeps the CI integration test fast (<5s) and isolated from data updates. The real fixture is tested by the same loader; if the loader works on the sample, it works on the real one.
- Idempotency assertion = the difference between "we say it's idempotent in the README" and "we know it's idempotent because the test fails otherwise".

**Rejected:** Running live FCBQ scrape in CI (slow, fragile, and an actual ToS risk if CI runs frequently), forbidding `ingest.*` via static analysis tools (overkill — grep is enough).

### D2.5-11 — ADR-0005: offline batch FCBQ ingest (mandatory deliverable)
**Decision:** ADR-0005 committed at `docs/adr/0005-offline-batch-fcbq-ingest.md` before Phase 2.5 ships, documenting:

- **Context:** SC6 from P2 deferred — need real Catalan data in prod, can't violate LOCKED no-live-ingest.
- **Decision:** offline CLI scrapes FCBQ → JSON fixtures (committed) → loader.
- **Alternatives considered:**
  - **A) Live fetch endpoint** (FastAPI calls FCBQ per request). Rejected: violates LOCKED constraint, ties API uptime to FCBQ uptime, slow responses (1-3s vs <50ms), and easy ToS violation if traffic grows.
  - **B) Cron worker on Render hitting FCBQ daily.** Rejected: still introduces FCBQ→prod runtime coupling, needs separate worker dyno billing, adds deploy complexity. Acceptable in v2 once auth/writes (P3) make the API genuinely live.
  - **C) Materialized cache (Redis updated by cron).** Rejected: re-introduces Redis (dropped 2026-05-19 as dead-weight); just moves the live-fetch problem one hop.
  - **D) Continue with `seed/minimal.py` fictitious data.** Rejected: fictional names fail the "real Catalan basketball" recruiter signal, and Catalan basketball is the entire point of differentiating from yet-another-NBA-clone portfolio.
- **Consequences:**
  - **+** Prod runtime stays pure (zero external HTTP). API uptime independent of FCBQ.
  - **+** Fixtures versioned in git — fully diff-able, replayable, auditable provenance.
  - **+** CI can replay any fixture without HTTP — fast + deterministic tests.
  - **+** No API keys / secrets needed.
  - **−** Data goes stale until next manual scrape. Acceptable: this is a portfolio API, not a live product. Users (recruiters) don't need real-time game scores.
  - **−** FCBQ HTML changes break the CLI silently if not noticed. Mitigated by `FCBQParseError` with raw-path pointer + CI sample fixture catches schema regressions.
- **Future:** A v2 cron worker becomes viable after P3 (auth+writes) — only then is the API "live enough" to warrant background refresh.

**Why this ADR is recruiter-critical:** It's the doc a senior engineer opens FIRST when reviewing the project. It demonstrates "this person thought about trade-offs, didn't just pick the obvious thing, and wrote the reasoning down." ADR-0003 (VAL formula) and ADR-0004 (tie-breakers) are technical; ADR-0005 is **architectural** — and architectural ADRs are what gets quoted in technical interviews.
</decisions>
